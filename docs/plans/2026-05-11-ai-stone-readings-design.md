# Design: AI Stone Readings

**Date:** 2026-05-11 · **Author:** Anastasia (brainstormed with Claude) · **Status:** Approved 2026-05-11 · **Branch:** `feature-ai-readings`

> Bonus feature on top of Lab 3. Adds two LLM-generated readings — a perfume
> pairing and a mystical personality description — both keyed on the stones
> the logged-in user picks. The graded `lab3` branch is **not** modified.

---

## 1. Goal

Let a logged-in user pick 1–3 stones from the encyclopedia and receive either:

- a **perfume pairing** tailored to the stones and their registered gender, or
- a **personality reading** woven across the chosen stones.

Both readings are generated by Google Gemini 1.5 Flash at `temperature=0`,
deduplicated by input tuple, and saved to the user's profile history.

## 2. Non-goals

- No anonymous-user access — readings need a user profile.
- No mid-flow gender quiz — gender is captured once on registration (and is
  optional with a "prefer not to say" opt-out).
- No streaming responses — the call is one-shot; we display a spinner during
  the 2–5 s round trip.
- No real-time perfume catalog verification — we accept the small risk of
  LLM-hallucinated perfume names; `temperature=0` minimises it. UI shows a
  small note: *"AI-generated suggestions; verify before buying."*

## 3. User flow

1. A logged-in user sees a new **"My reading"** link in the top nav (between
   *Home* and the role badge). Anonymous visitors never see it.
2. Clicking opens `/my-reading` (route protected with the existing
   `ProtectedRoute`; anonymous → redirect to `/login`).
3. The page shows:
   - A two-button **mode toggle**: *Perfume* / *Personality*. Default Perfume.
   - A **grid of 10 stone cards** (same thumbnails as the homepage). Click to
     select; click again to deselect. Cap of 3 selected. Selecting a 4th does
     nothing (with a small "max 3" hint).
   - A **Generate** button, disabled while no stones are selected or while a
     request is in flight.
4. On click → 2–5 s spinner → result renders below, also saved to history.
5. The user's profile page (`/users/:id`) gains a **"My readings"** section:
   a chronological list (newest first, paginated at 20) of past readings with
   mode badge + chosen stones + date. Click an item to expand its content.

If the user's `gender` is NULL or `prefer_not_to_say`, perfume mode silently
restricts itself to unisex fragrances. No nag modal.

## 4. Data model

**Schema migration applied to Supabase Postgres:**

```sql
-- 1. Add gender to existing users table (nullable, opt-in)
ALTER TABLE users
    ADD COLUMN gender VARCHAR(20) NULL
    CHECK (gender IS NULL OR gender IN ('female', 'male', 'prefer_not_to_say'));

-- 2. New table for the readings
CREATE TABLE ai_readings (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    kind            VARCHAR(20) NOT NULL CHECK (kind IN ('perfume','personality')),
    stone_slugs     TEXT NOT NULL,                  -- sorted, comma-joined
    gender_at_time  VARCHAR(20) NULL,               -- snapshot for dedupe stability
    content         TEXT NOT NULL,                  -- LLM-generated markdown
    created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    UNIQUE (user_id, kind, stone_slugs, gender_at_time)
);

CREATE INDEX idx_ai_readings_user_created ON ai_readings (user_id, created_at DESC);
```

### Dedupe logic

Because `temperature = 0` makes Gemini deterministic, the same
`(user, kind, stones, gender)` always yields the same output. The
`UNIQUE` constraint encodes this — the backend checks for a matching row
*before* calling Gemini and returns the cached one on hit. This:

- Eliminates duplicate API spend.
- Keeps the user's history clean (no doppelganger rows).
- Lets us treat the read endpoint as authoritative history.

The `gender_at_time` snapshot means a user changing their registered gender
later doesn't silently invalidate previous readings.

## 5. API endpoints

| Method | Path | Body / params | Auth | Returns |
|---|---|---|---|---|
| `POST` | `/ai/readings` | `{ kind: "perfume"\|"personality", stone_slugs: string[] }` (1–3 slugs) | JWT (any role) | `{ id, kind, stone_slugs, gender_at_time, content, created_at }` |
| `GET` | `/ai/readings` | `?limit=20` (default 20, max 50) | JWT (any role) | `[{ id, kind, stone_slugs, content, created_at }, ...]` newest first |

### Validation

- `kind` must be one of the two literals.
- `stone_slugs` must be a list of 1–3 unique existing article slugs.
  Backend normalises by sorting + lowercasing before lookup.
- Returns `400` on schema violations, `401` without JWT, `404` on unknown slug.

### Error responses

| Trigger | Status | Body |
|---|---|---|
| `GEMINI_API_KEY` env var missing | 503 | `{detail: "AI features not configured"}` |
| Gemini rate limit (429) | 429 | `{detail: "AI rate limit — try again in a moment"}` |
| Gemini 5xx or timeout | 502 | `{detail: "AI service unavailable"}` |

## 6. LLM prompts

Both prompts are stored as constants in `app/services/gemini.py`.

### Perfume mode

**System:**
> *"You are a thoughtful perfume guide. Given the user's chosen stones and
> gender, suggest a fragrance pairing. If the user's gender is
> 'prefer_not_to_say' or unset, recommend only unisex perfumes. Output exactly
> two parts: (1) a 2–3 sentence opening paragraph describing the overall
> scent profile that fits the stones, then (2) a markdown bullet list of
> exactly two real perfume picks in the format `**Brand — Name** — one
> sentence on why it fits.` Do not invent perfumes; if unsure, fall back to
> classics. Be warm but concise."*

**User template:**
> *"Stones: {sorted, comma-joined slugs}. Gender: {gender_or_unspecified}."*

### Personality mode

**System:**
> *"You are a writer of mystical, warm-toned personality readings. Output
> exactly three markdown sections in this order: `## Your inner climate`,
> `## What others see in you`, `## What these stones are asking of you`. Each
> section is one paragraph of roughly 50–70 words. Weave together all the
> chosen stones; do not list them separately. Be poetic but specific."*

**User template:**
> *"Stones: {sorted, comma-joined slugs}."*

### Generation parameters

- `model = "gemini-1.5-flash"`
- `temperature = 0`
- `max_output_tokens = 600`
- `top_p = 1`

### Markdown rendering

The LLM emits markdown. We render with `marked` (5 KB gzipped, no React
dependency) into plain HTML, then drop it into a `<div>` with
`dangerouslySetInnerHTML`. Safe because (a) the markdown comes from our own
server which got it from Gemini (no untrusted user input), and (b) `marked`
escapes raw HTML by default in the version we'll pin.

### Loading & error UX

- Generate button shows spinner inside itself + becomes disabled while a
  request is in flight.
- On 503/429/502, an `<Alert>` appears above the result panel with a Retry
  button.

## 9. Existing-user handling

`admin` and `regular` were seeded before this feature. Their `gender` column
is NULL. The backend treats NULL identically to `prefer_not_to_say` → perfume
prompt asks for unisex picks. No data migration needed for existing users.

If they want to set their gender, they can use the standard edit-user flow
(`/users/:id/edit`) which we extend with the gender field.

- PRs follow the existing flow: `feature-ai-readings → development → main`,

- Vercel auto-deploys from `main`. Render auto-deploys from `main` after the

- The Supabase migration runs **once** before the backend deploy — via the
  Supabase dashboard SQL editor, by Anastasia. The backend is forward-

  work; only `/ai/*` fails.

## 12. Out-of-scope (deferred to future labs / iterations)

- **Streaming responses** — would feel snappier but adds significant
  complexity; skipping.
- **Per-stone deep links** ("Get a reading for *this* stone") — could be added
  later as a one-click shortcut on the article page.
- **Sharing readings publicly** — requires public read URLs + safe content
  moderation; defer.
- **Multi-language output** — Gemini handles it, but UI isn't translated;
  keep readings in English for now.
- **Cross-stone collections / wishlists** — separate concern; not part of
  this design.
